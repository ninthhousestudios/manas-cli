# manas-cli — boot contract

Status: draft (v0)
Date: 2026-05-04
Owner: manas-cli
Gates: phase-1 manas-cli implementation. No adapter or skill-shell code lands until this doc + its acceptance test exist.

This doc specifies how an LLM harness gets connected to manas's tool surface, how that connection is scoped, and what happens when scoping fails. Everything else in manas-cli (adapters, skill shells, sideband) builds on this.

It is the load-bearing **hard** contract that enforces principle 11 (harness-agnostic), principle 9 (manas-cli owns cross-tier composition), and the boot-mode distinction in `docs/manas-architecture.md` (`minimal` vs `rich`). CLAUDE.md is **not** part of this contract; it is configuration-as-prose and may or may not be honored by the LLM. The boot contract is what survives the LLM ignoring CLAUDE.md.

---

## 1. actors

| Actor | Role |
|---|---|
| **operator** (human) | Starts a session by invoking `manas <subcommand>` or by launching a harness directly. |
| **manas-cli** | Creates the per-session binding, launches the harness with the right config, tears the binding down on exit. The trusted boot agent. |
| **harness** | Claude Code, Gemini CLI, Codex CLI, opencode, or any future MCP client. Receives a single MCP endpoint and an access token. Has no awareness of subsystems. |
| **mcpjungle** | The MCP gateway. Runs as a long-lived daemon. Owns Tool Groups, MCP clients, and the ACL. The only enforcement point. |
| **subsystem servers** | chitta, smriti, sutra, sangha, yojana (and later kosha). Registered with mcpjungle as upstream MCP servers. Never reachable by the harness directly. |

**Trust boundary:** the operator and manas-cli are trusted. mcpjungle is trusted. The harness and the LLM running in it are *not* trusted to honor any soft instruction; the boot contract must hold even if the LLM is adversarial within its tool-call surface.

## 2. boot modes (recap)

Two named modes. Each maps to a Tool Group on mcpjungle.

| Mode | Tool Group | Reachable subsystems | Default for |
|---|---|---|---|
| **minimal** | `code` (or `none` for pure editing) | sutra only (or nothing) | Blind code review, fresh-eyes work, untrusted contexts |
| **rich** | `full` | chitta, smriti, sutra, sangha, yojana, (kosha) | Continuity work — handoff, /reflect, dogfood sessions |

Tool Groups are defined in mcpjungle (`internal/model/tool_group.go`). The Tool Group → effective-tools resolution is mcpjungle's job; manas-cli only references Tool Groups by name.

A binding selects exactly one Tool Group. Mode changes require a new session (a new binding). There is no in-session escalation.

## 3. the binding — what the harness receives

A **boot binding** is a tuple, materialized as env vars and a config file written by manas-cli before the harness launches.

| Field | Source | Lifetime | Notes |
|---|---|---|---|
| `MANAS_MCP_ENDPOINT` | mcpjungle's group-specific MCP URL (e.g. `http://127.0.0.1:8080/v0/groups/code/mcp`) | session | **Mode-specific.** Each boot mode maps to a different group endpoint. The Tool Group is determined by the URL, not the access token. See OQ-1 resolution. |
| `MANAS_MCP_TOKEN` | One-shot bearer token issued by mcpjungle for this session | session | Bound to a single mcpjungle `McpClient` row whose ACL maps to the chosen Tool Group. Never reused across sessions. |
| `MANAS_SESSION_ID` | UUIDv7 generated by manas-cli at boot | session | Used to correlate sangha session, chitta session_summary, yojana audit, transcript path. |
| `MANAS_TRANSCRIPT_PATH` | Resolved by the harness adapter | session | Adapter-specific. Skill shells inject this into skill bodies that need it (kills the hardcoded `~/.claude/projects/...` in `/done`). |
| `MANAS_PROJECT_ROOT` | `cwd` at `manas` invocation, canonicalized | session | Sangha and yojana scoping. |
| `MANAS_BOOT_MODE` | `minimal` \| `rich` | session | Informational; ACL is the truth. Useful for the harness to render UI hints and for skills to skip rich-only steps in minimal mode. |

The harness adapter is responsible for translating these env vars into the harness's native config (e.g. CC's `.mcp.json`, opencode's config, a Gemini CLI flag). What the harness sees is **a single MCP endpoint**. It cannot see the access token's scope, and it cannot enumerate subsystems it isn't entitled to.

**Hard rule:** the harness never receives subsystem URLs, subsystem credentials, or per-server MCP configs. The only MCP endpoint it knows about is mcpjungle's.

## 4. lifecycle

### 4.1 create

Triggered by `manas <subcommand>` or `manas warm` / `manas dev`. Steps, in order:

1. **Health gate.** manas-cli calls mcpjungle health + each upstream subsystem's `*_health` (cheap calls). If any required-for-this-mode subsystem is down, fail loud (§5).
2. **Tool Group lookup.** Resolve the requested mode → Tool Group name. Verify it exists on mcpjungle. If absent, fail loud.
3. **Mint a session token.** manas-cli calls mcpjungle's admin API to create a fresh `McpClient` (or its successor — see §8 OQ-1) with name `manas-session-<MANAS_SESSION_ID>`, mapped to the chosen Tool Group. mcpjungle returns the access token.
4. **Register sangha session.** Connection-bound; happens via the harness on first MCP call. manas-cli does *not* pre-register; sangha's identity is bound to whoever connects with the token. (Note: this means sangha session_register must be in every Tool Group, including `code`, or `code` skips sangha entirely. See §8 OQ-2.)
5. **Write env + config.** Compose the env vars in §3, write any harness-specific config files into a per-session scratch dir (`~/.manas/sessions/<MANAS_SESSION_ID>/`).
6. **Launch the harness.** Adapter-specific. Foreground process; manas-cli waits on it.
7. **Record the binding.** Append a row to `~/.manas/bindings.log` (jsonl): session id, mode, tool group, mcpjungle client name, harness, pid, start_at. Used by `manas status`, `manas done`, and the rotation/revoke paths.

### 4.2 rotate

There is no rotation in v0. A binding lives for the lifetime of one harness process. If the operator wants a different mode, they exit and re-`manas`.

(This is a deliberate scope cut. Rotation requires the harness to refresh credentials mid-session, which CC/Gemini/Codex/opencode handle differently and inconsistently. We'll add it when a real workflow demands it.)

### 4.3 teardown

On harness exit (clean or crash):

1. **Revoke the token.** manas-cli calls mcpjungle's delete-client API for `manas-session-<MANAS_SESSION_ID>`. The token is now dead even if it leaked.
2. **Release sangha resources.** manas-cli calls `session_unregister` (idempotent). Locks held by the session expire on TTL; we don't try to enumerate-and-release.
3. **Mark the binding row** in `bindings.log` with `end_at` and exit status.
4. **Scratch dir cleanup.** Configurable retention; default keep last 10 sessions for debugging.

If manas-cli itself dies mid-session, the harness's token outlives it. Recovery: `manas gc` scans `bindings.log` for rows with no `end_at` and a missing pid, revokes the token, and marks them. Run by `manas health` opportunistically. **This is a known gap; it's secure-degraded, not prohibited, because the token still scopes to the original Tool Group.**

## 5. failure semantics

Per the table in `docs/manas-architecture.md`, classified.

| Failing component | At boot | Class |
|---|---|---|
| mcpjungle down | Refuse boot. Loud error pointing at `manas health`. **Never** fall back to per-server MCP. | prohibited |
| mcpjungle reachable but Tool Group not found | Refuse boot. Likely operator error or stale config. | prohibited |
| Required subsystem down (e.g. chitta in `rich` mode) | Refuse boot for `rich`. Allow `minimal` if its required subsystems are up. | secure-degraded |
| Optional subsystem down | Continue. Log warning. The agent will get a tool-call error if it tries to use it; that's acceptable. | secure-degraded |
| Token mint succeeds, harness launch fails | Revoke the token, clean up the scratch dir, surface the harness error. | secure-degraded |

**Emergency mode.** `manas dev --no-gateway` (declared in arch doc, not implemented in v0): boot the harness with per-subsystem MCP configs, no Tool Group ACL. Banner on every command output that ACL is off. Every tool call logs the bypass to `~/.manas/insecure-emergency.log`. Refuses to start if a sangha session is registered for the project (so it's never the path of least resistance during normal work). **Not part of v0; specified here so the design space is closed.**

## 6. acceptance test (gates phase 1)

Single integration test. Must pass before any phase-1 manas-cli code merges.

**Setup:**
- mcpjungle running with Tool Groups `code`, `full`, `none` defined.
- chitta, smriti, sutra, sangha, yojana running and registered as upstream servers.
- A fixture harness — a synthetic MCP client we control — playing the role of "adversarial agent."

**Test 1 — minimal boot blocks privileged tools.**
1. `manas-cli` boots a session in `minimal` mode (Tool Group = `code`). Returns binding to fixture harness.
2. Fixture harness connects to `MANAS_MCP_ENDPOINT` with `MANAS_MCP_TOKEN`.
3. Fixture harness calls `chitta__store_memory` by exact name. **Expected:** mcpjungle returns "tool not found" or "forbidden". The call never reaches chitta.
4. Fixture harness calls `smriti__smriti_read`. **Expected:** same.
5. Fixture harness calls `sangha__session_register`. **Expected:** depends on §8 OQ-2; the test pins the answer.
6. Fixture harness calls `sutra__sutra_health`. **Expected:** success. Proves the binding works for what it should.

**Test 2 — rich boot allows everything; no leakage across sessions.**
1. `manas-cli` boots session A in `rich`. Session B in `minimal`. Two separate fixture harnesses.
2. B's token is rejected for chitta calls (re-runs Test 1 with B). A's token works.
3. Session A exits. mcpjungle teardown runs.
4. Session B (still active) replays A's old token. **Expected:** rejected (token is revoked).

**Test 3 — orphaned binding GC.**
1. Boot a session, kill manas-cli with SIGKILL while harness is still up.
2. `manas health` (or `manas gc`) detects the orphan, revokes the token.
3. Harness's next call fails with auth error.

The test is the contract. If a future change breaks any assertion, that change either updates the contract or doesn't ship.

## 7. what this contract does not specify

- **Skill-shell behavior.** Lock claim, env injection, body invocation. Specified separately as part of phase-1 manas-cli design. The boot contract gives skill shells the env vars they need (`MANAS_SESSION_ID`, `MANAS_TRANSCRIPT_PATH`, `MANAS_PROJECT_ROOT`).
- **Harness adapter internals.** Each adapter handles its own config translation; they share the binding shape from §3.
- **Sideband daemon protocol.** The Unix socket and its endpoints are out of scope. The sideband is a separate trust boundary (manas-cli ↔ subsystems, no harness involvement).
- **The compound-tool host.** Cross-tier compound operations (yojana_context resolution, darshana joins) are surfaced by manas-cli to the harness as additional MCP tools fronted by mcpjungle. They get the same Tool Group treatment as any other tool. Their internal routing is design-time, not boot-time.

## 8. open questions

These don't block landing the contract, but each needs an answer before the corresponding code lands.

**OQ-1 — mcpjungle's per-session client API.** RESOLVED — answer is (c). See `manas-cli/docs/spikes/oq-1-mcpjungle-binding.md`.

Each Tool Group is its own MCP endpoint (`/v0/groups/<name>/mcp`). Scoping is endpoint selection, not client binding. No mcpjungle PR is needed for v0. Two corrections to this document follow from the spike:

1. `MANAS_MCP_ENDPOINT` must be **mode-specific** (e.g. `http://127.0.0.1:8080/v0/groups/code/mcp`), not a single shared URL. The note in §3 that "same endpoint regardless of mode" is wrong.
2. "The Tool Group is determined by the access token, not the URL" (§3) is backwards — it is determined by the URL. The token is auth-only in the group-endpoint path; mcpjungle does not enforce which group a given token may access.

**OQ-2 — should `code` (minimal) include sangha?** Sangha's `session_register` and presence are useful even for blind reviews (knowing another agent is in the project). But every tool added to `code` widens the minimal surface. Decision needed before defining the canonical Tool Groups. Lean: include `sangha_session_register`, `sangha_session_heartbeat`, `sangha_session_unregister`, `sangha_read_inbox`. Exclude `resource_claim` / `resource_release` / `broadcast` (those are for cooperating sessions, not blind ones).

**OQ-3 — Codex CLI MCP support.** ~~Verify in the headless-spike step (sequence step 2): can Codex CLI consume an MCP endpoint headlessly with a bearer token? If not, the adapter for Codex is a no-tools shell — skill bodies running under Codex have no tool surface and must be designed accordingly.~~

**Resolved (2026-05-04).** All four harnesses can run tool-calling headlessly with an HTTP MCP endpoint and bearer token. No no-tools fallback is needed.

- **Claude Code**: `claude -p` + `--mcp-config` JSON file with `type: "http"` and `headers`. Use `--strict-mcp-config` to lock the surface.
- **Gemini CLI**: `gemini -p` + `.gemini/settings.json` with `type: "http"` and `headers`. Requires `GEMINI_CLI_TRUST_WORKSPACE=true` env var for headless runs.
- **Codex CLI**: `codex exec < /dev/null` + `CODEX_HOME` isolation with config.toml `[mcp_servers.X]`. Bearer token via `bearer_token_env_var` (env var ref only — token never written to disk; best security of the four).
- **opencode**: requires two-process pattern (`opencode serve` + `opencode run --attach`). Works but has adapter lifecycle complexity. Deferred to v2.

Smoke tests passed for all four: yojana_query called and returned correct results; chitta health_check called through bearer auth. Evidence and full findings in `manas-cli/docs/spikes/headless-harness-matrix.md`.

**OQ-4 — token storage in the harness.** Some harnesses persist MCP server config to disk (CC writes `.mcp.json`). The token leaks to disk for the session's lifetime. Acceptable for v0 (the token is single-session and revoked on exit), but worth flagging if a multi-session reuse pattern emerges.

**OQ-5 — concurrent sessions per project.** Two `manas` invocations in the same project: each gets its own session id, its own token, its own sangha session. Contention is handled by sangha locks (per principle 6, hard contract). No special boot handling. Confirm this works end-to-end in test 2 above; if not, add a fourth test.
