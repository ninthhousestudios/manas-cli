## phase 0 â€” design decisions before code

These shape data formats and APIs. Cheap now, expensive after code lands.

### 0.1 mental-model retirement protocol

**Problem:** `/reflect` creates mental models. Models go wrong. "Tag, don't delete" preserves them, but a wrong model pollutes every future search result. Current system has `superseded_by` in metadata but no `retired_at`, `contradicts`, or trust signal.

**Design:**

- `update_memory` gains the ability to set `metadata.retired_at` (ISO timestamp) and `metadata.retirement_reason` (free text).
- Search gains an optional `exclude_retired: true` parameter (default true). Retired models are excluded from normal search but still retrievable by ID.
- `/reflect` checks for contradictions between new observations and existing models. When found, it either updates the model or retires it with a reason.
- Trust signal deferred â€” "how long since this model was last confirmed" is useful but complex. Revisit after /reflect has been running for a while and we see how models age in practice.

### 0.2 freshness envelope

**Problem:** Every read tool returns results confidently when they may be stale. Qartez index could be hours old. Chitta results are always fresh (Postgres), but smriti results depend on scan recency.

**Design:** see `manas/docs/freshness-envelopes.md` for the cross-subsystem rule and per-subsystem `as_of` / `is_stale` semantics. Conformance tasks tracked in each subsystem's todo/handoff.

## phase 1 â€” sangha (session coordination)

**Goal:** Multi-session safety. Solve the concurrent `/done` and `/reflect` problem before they cause real damage.

**Why first:** Sangha is small (~1500 LOC Rust), independent of smriti and mcpjungle, and solves an active problem. Two CC sessions both running `/done` silently lose a handoff today. Building sangha first means `/done` and `/reflect` become safe before we build more complex subsystems on top of them.

**Reference:** `chitta/docs/claude-presence-rewrite.md`. Rust rewrite of `claude-presence` (MIT, ~1100 LOC TypeScript). Preserves upstream design, adds: user-scope locks, canonical lock vocabulary, connection-bound identity, auto-extend locks on heartbeat.

### deliverables

1. **Repo scaffold.** New repo `~/soft/sangha`. Rust, Cargo.toml. ~8 source files + tests.
2. **Core: sessions + locks.** SQLite-backed session registry with heartbeat TTL. Advisory resource locks with per-claim TTL, user-scope via `__user__` sentinel, `long_op` flag for extended operations.
3. **Inbox.** Broadcast messages between overlapping sessions. Advisory only â€” document it as "for humans, not coordination."
4. **Connection-bound identity.** First `session_register` binds identity to the MCP connection. Prevents trivial impersonation within a single process.
5. **MCP tools (9).** `session_register`, `session_heartbeat`, `session_unregister`, `session_list`, `resource_claim`, `resource_release`, `resource_list`, `broadcast`, `read_inbox`.
6. **CLI surface.** `sangha status`, `sangha locks`, `sangha clear`, `sangha path`.
7. **Hooks.** SessionStart auto-registers. UserPromptSubmit surfaces presence context as a one-liner. Both degrade silently if sangha binary is missing.
8. **Skill updates.** `/done` claims `handoff` lock before writing handoff.md, releases after. `/reflect` claims `reflect:user` (user-scope, long_op), skips if held.

### canonical lock vocabulary

| Resource | Scope | Default TTL | Held by |
|---|---|---|---|
| `handoff` | project | 120 s | `/done` while writing `docs/handoff.md` |
| `reflect:user` | user | 600 s | `/reflect` for the entire run |
| `smriti-scan` | user | 300 s | smriti daemon during scan |

### order of implementation

```
schema + db â†’ sessions + locks â†’ identity â†’ MCP tools â†’ CLI â†’ inbox â†’ hooks â†’ skill updates
```

Tests first for identity binding and TTL auto-extension â€” those are the two behaviors that diverge from upstream.

## phase 2 â€” smriti v0.1

**Goal:** Backup audit + filesystem perception for the agent.

**Prerequisite:** Phase 0 decisions baked into design. Specifically 0.1 (retirement protocol) and 0.2 (freshness envelope) since smriti's schema embeds both.

### deliverables

1. **Repo scaffold.** New repo `~/soft/smriti`. Rust, Cargo.toml. ~12 source files per the sketch.
2. **Ignore parser.** `.smritiignore` with `[catalog]` and `[no-embed]` sections. Hardened defaults at `~/.smriti/ignore`. Gitignore-compatible glob matching.
3. **Scanner.** Walk allowlisted roots, classify tiers, mtime+size short-circuit, blake3 hashing, event emission, move detection (fuzzy filename fallback), hardlink/symlink handling.
4. **SQLite storage.** Schema with sqlite-vec + FTS5. Documents, paths, events, catalog, snapshots tables.
5. **MCP tools.** `smriti_scan`, `smriti_find`, `smriti_get`, `smriti_read`, `smriti_map`, `smriti_outline`, `smriti_history`, `smriti_audit`, `smriti_manifest`, `smriti_health`. All with freshness envelope.
6. **Daemon.** Long-lived process, Unix socket, async scan, systemd user unit.
7. **CLI surface.** `smriti scan`, `smriti audit`, `smriti roots add/list/remove`, `smriti init`.
8. **Privacy gate.** Allowlist enforcement in every read path. Read audit log. `smriti_read` as the gated content accessor.
9. **Embedding (optional leg).** BGE-M3 ONNX, gated on `SMRITI_MODEL_PATH` + `[no-embed]` filter. Degrades to BM25-only when absent.

### order of implementation

Build scanner + storage first (can be tested without MCP). Then MCP tools. Then daemon wrapping.

```
ignore parser â†’ scanner â†’ storage â†’ MCP tools â†’ daemon â†’ CLI â†’ embedding
```

The first useful milestone is `smriti scan` + `smriti audit` working from the CLI â€” the backup audit use case, no MCP needed.

## phase 3 â€” mcpjungle integration

**Goal:** Single MCP endpoint fronting chitta, sangha, qartez, smriti.

**Why now (after smriti v0.1):** mcpjungle is most useful when there are 3+ MCP servers to coordinate. With chitta + sangha + qartez + smriti, the gateway pays for itself. Before that it's ops overhead without enough win.

### deliverables

1. **mcpjungle deployment.** Go binary, systemd user unit, config for chitta + sangha + qartez + smriti upstreams.
2. **Tool Groups.** Define groups: `memory` (chitta tools), `code` (qartez tools), `filesystem` (smriti tools), `presence` (sangha tools), `full` (all). Blind code-review sessions get `code` only.
3. **CLAUDE.md update.** Single MCP endpoint in config. Remove per-server MCP configuration.
4. **Permission model via Tool Groups.** Per-skill scoping. `/reflect` gets `memory` + `filesystem` + `presence`. Code review sessions get `code` only.

### known gaps to work around

- No resource templates in mcpjungle â€” smriti uses tool-based reads (already designed this way).
- No resource subscriptions â€” change notifications via manas-cli sideband.
- Tool Groups are tools-only â€” resources not group-scoped. Accept for now.

## phase 4 â€” chitta v0.0.4 + manas-cli

**Goal:** Chitta-side improvements identified in phase 0, plus the human-facing ops surface.

### chitta v0.0.4

1. **Mental-model retirement.** `metadata.retired_at`, `metadata.retirement_reason`. Search excludes retired by default. (Phase 0.1 design.)
2. **Freshness on health_check.** Minor addition â€” report operational freshness signals.
3. **Sideband hook for smriti.** When smriti detects a file move, chitta can be notified to update any memories that reference that path in metadata. Scoped narrowly.

### manas-cli

1. **`manas warm`** â€” rich boot. Read handoff, load recent mental models, check all subsystem health, surface sangha inbox messages from prior overlapping sessions.
2. **`manas done`** â€” session shutdown. Calls chitta `/done` (which claims sangha `handoff` lock), enqueues smriti rescan.
3. **`manas reflect`** â€” between-session maintenance. Claims sangha `reflect:user` lock, calls chitta `/reflect`.
4. **`manas health`** â€” all-subsystem health check in one command. Includes sangha: "N active sessions, M held locks."
5. **CLAUDE.md fragment templating.** Each subsystem provides a CLAUDE.md fragment. `manas init` composes the project-level CLAUDE.md from fragments.
6. **Sideband daemon.** Runs alongside mcpjungle. Handles smritiâ†’chitta path-move notifications and other narrow cross-subsystem coordination that shouldn't go through the LLM.

## principles (from opus 4.7 review, accepted)

1. **Default-deny for smriti.** Allowlisted roots, hardened ignore defaults, embedding gate.
2. **Freshness envelope on every read.** `as_of` + `is_stale`. System-wide.
3. **Hash whole content.** No frontmatter stripping. `minor_change` event for metadata-only edits.
4. **smriti_read is the privacy gate.** Built-in filesystem reads bypass policy.
5. **Daemon transport for smriti.** Not stdio-per-CC.
6. **stdio transport for sangha.** No daemon needed â€” SQLite handles concurrency, no long-running work.
7. **sqlite-vec + FTS5 for smriti.** No tantivy, no separate index files.
8. **mcpjungle as gateway.** MPL-2.0. Don't write our own.
9. **Narrow sidebands only.** smritiâ†’chitta path-move sync through manas-cli daemon. No arbitrary subsystem calls.
